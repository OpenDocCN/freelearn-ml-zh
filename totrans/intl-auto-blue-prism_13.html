Other Intelligent Blue Prism Products and Future IA Trends

Outside of the core BP RPA software, there are other products that round out BP’s IA portfolio. In this chapter, we’ll discuss four of these products. Our goals for this chapter are to understand the products’ purpose and to see how they fit into IA. We’ll also walk through the high-level steps of using each product. Finally, we’ll see what next steps are needed if you’d like to try them out and how to find out more information. To finish off the book, I’ll also discuss what I think are the future trends of IA.

The four BP products that we’ll discuss are the following:

  • Decipher Intelligent Document Processing (IDP)
  • Document Automation
  • Decision
  • Interact

Decipher IDP

Decipher IDP is a document-processing solution that provides OCR, document classification, entity extraction, table extraction, and a web interface for HITL correction and training of documents. Decipher was released in June 2020 and has received continuous upgrades since, so it has been battle-tested by many other firms. If you need to extract content from scans or images of computer-generated (not handwritten) documents, you should consider using Decipher.

As someone who has used Decipher, and has heard feedback from BP customers who’ve used it, it’s definitely worth considering for computer-generated documents. It also works well for documents with tables that have varying numbers of rows. Forms with lots of handwriting aren’t supported yet (although it is supported in the next product that will be discussed).

I’ve seen some BP customers and partners build less fully featured document-processing solutions in-house, which takes months of development effort at a minimum. Decipher gives you a document-processing solution free of charge, provided that your company has either a Production or Business-Critical support plan with BP.

How is Decipher related to IA?

Extracting fields from documents so that they can be processed by RPA is one of the most common IA use cases, so much so, that every major RPA vendor has a product dedicated to that task.

Behind the scenes, Decipher uses ML in multiple places. First, Decipher uses an OCR engine (Tesseract), which itself has pretrained ML models that are used to recognize text. Next, Decipher can train document classification models to differentiate between different types of documents. For example, you might have an email account that receives both quotations and invoices as attachments. Decipher can help you differentiate between these two types of documents and start them down different automation processing paths.

Decipher can also train document-specific models to extract pre-specified entities, or structured data from documents. Pre-specified means that you have to tell Decipher ahead of time what you’re trying to extract, for example, an invoice number or a total price. When you’re presented with the web UI to correct the entity predictions, you’ll see exactly the fields that you’ve specified. Decipher can also handle semi-structured data, meaning documents with fields in fixed positions, as well as tabular items that can vary in length.

Finally, there’s also an NLP plugin that can extract unstructured data. This plugin requires a GPU to be installed. From looking at the third-party license acknowledgments of the plugin, it uses Keras and Tensorflow, indicating that some neural networks or deep learning models are being used here.

Decipher also comes with a HITL web interface that can be used to verify and correct predictions for extracted document fields. The extracted fields for your pre-specified entities are displayed side-by-side to the actual document so that it’s easy to compare the two and make amendments.

Using Decipher

A high-level view of how you’d use Decipher is the following:

  1. In the Decipher web interface, define what types of documents you want to extract, such as application forms, invoices, financial statements, etc.
  2. For each document type that was previously defined, specify which fields and tables you want to extract. These first two steps are only done once, unless you need to change what fields you want to extract from the document. Changing the fields will result in having to retrain the entire ML model for this document type.
  3. From BP, send in a batch of images or documents into Decipher. This is done through a sequence of Actions from the Decipher VBO: Create Batch, Add File to Batch (which must be done on each individual file) and Submit Batch.
  4. Decipher processes the documents in the background and tries to perform the classification or extraction. It determines whether human review of documents is necessary based on the confidence score from the model and the default threshold values, which can be changed by the user.
  5. The user checks to see whether there are documents that need to be reviewed. This checking can be done either from a BP Process, by using the Has Batches to Verify Action, or by visiting the Decipher UI and checking manually. If HITL review is required, the user visits the web UI and corrects what Decipher has extracted. Corrections made will be fed back into the ML model, improving the accuracy over time.
  6. In BP, there needs to be a Process that checks whether a completed batch exists, using the Decipher VBO Action Get Next Completed Batch. A completed batch contains human-reviewed documents, and documents that didn’t need human review due to surpassing the threshold requirements. If there are completed batches, we can loop through the completed batches’ documents extracted data for BP processing using the Get Document Data Action.

Next steps

Decipher can be directly downloaded from the BP Portal: https://portal.blueprism.com/products/decipher. If you don’t have a Decipher license already, you’ll need to request one from your sales representative.

You’ll also need to provision new infrastructure to install Decipher on. The minimum requirements are listed here: https://bpdocs.blueprism.com/decipher-2-2/en-us/getting-started/minimum-requirements.htm. An objection that I’ve seen customers raise with using IDP comes from the infrastructure requirements and especially the added dependencies. Decipher requires a RabbitMQ message broker to be installed, and some IT teams aren’t prepared or willing to support that.

For pilots or testing, it’s feasible to have the entirety of Decipher, including all of its dependencies installed on a single VM. For a production deployment, you’ll probably need another four VMs, unless you’re willing to host some of the Decipher components together, and that’s not accounting for high availability and disaster recovery. The infrastructure costs of Decipher should be considered ahead of time. The sizing of the VMs is also tricky, as you need to estimate the number of pages per hour that you intend to put through the different Decipher components. A sizing guide can be found here: https://bpdocs.blueprism.com/decipher-2-2/en-us/Guides/Architecture/sizing.htm.

Decipher does require some training to use. The learning curve isn’t as steep as regular BP, but you should still expect to spend at least a week or two playing around with the system before you start to become productive. More information about Decipher can be found on the BP Documentation website: https://bpdocs.blueprism.com/decipher-2-2/en-us/home.htm. Finally, the BP Portal website has templates and instructions for use with Decipher: https://portal.blueprism.com/products/developer-jumpstart.

Document Automation

Document Automation (DA) is BP’s second document processing solution. DA was in development at SS&C (BP’s parent company), before they acquired BP. Post-acquisition, DA was integrated into the BP product line. At a high level, DA is used to extract structured data, or documents with a fixed format where the positions of the document fields don’t change. It excels at handling low-quality digital documents and can deal with handwritten text.

I’ve not used DA under a production scenario, but I have gone through the internal training materials. One of the key benefits of DA is that it’s a completely hosted solution that’s compliant with HIPAA, SOC 2, GDPR, etc. There are no infrastructure concerns, unlike Decipher. It’s also a proven system with many customers and millions of pages processed. If you’re concerned about sending your documents outside of your firm, you can specify data retention periods in DA, so that documents older than a certain date are purged from the system.

DA differs from Decipher IDP in the following ways:

  • DA specializes in extracting handwritten or printed data that’s in low-quality digital formats. Examples are scanned and faxed documents that have low accuracy when extracted with standard OCR technology. Decipher can’t handle handwritten data as of version 2.2.
  • DA focuses on structured documents whereas Decipher can deal with structured, semi-structured, and unstructured documents (with the NLP plugin). DA doesn’t handle documents that have tables with varying numbers of rows.
  • DA is a hosted service that can’t be installed on premises. Decipher is installed, hosted, and maintained by the customer.
  • DA has a separate licensing model that’s unrelated to BP’s license. It isn’t free of charge. Decipher is free for up to 4,000 pages per license per month, as long as you’re on either the Production or Business-Critical BP support plan.
  • DA can be used as a standalone product without BP. The link between DA and BP is through a connector asset on the DX: https://digitalexchange.blueprism.com/dx/entry/9648/solution/chorus-document-automation. Decipher integrates closely with BP and can’t be used without BP, unless you develop some major custom workarounds.
  • DA doesn’t train individual ML models for each type of document that you have. Decipher has a loop in the product that goes through training, correcting the extracted data (if necessary), retraining, and getting feedback. While DA has a web interface to review the extracted data, it’s only used to correct data. Corrections aren’t used to improve a model that’s unique to you.
  • From an end-user perspective, DA is easier to use because many ML concepts don’t need to be considered. Decipher requires training and has a longer lead time to get started.

How is Document Automation related to IA?

DA fits a similar use case to Decipher. You’d use it to convert documents into CSV/JSON formats so that they can be further processed by BP. The difference lies in what types of documents DA excels at processing, the use of generic ML models (as opposed to a specific model that’s fit to your documents), and the user interface. DA is used for documents with fixed formats that have been potentially filled in with handwriting.

Behind the scenes, DA has a proprietary AI algorithm that uses neural networks. This model was trained on a human-corrected dataset of over 1.2 billion datapoints. The average accuracy of this model is 96% across all of DA’s enterprise customers.

Using Document Automation

At a high level, the steps to use DA are as follows:

  1. In DA’s web interface, define the fields you want to extract from a document, for example, first name, last name, and address. Fields have a “name” and a data type, for instance, Medium Text, Select One, Signature, and so on.
  2. Define a template. A template represents a form that can span multiple pages. Upload examples of the template pages to DA in image, or PDF format. Once a template file is uploaded, it will be displayed in the web UI, alongside all of the fields that you’ve defined.
  3. Draw boxes on the template image where your selected fields should appear. This is how you map which fields belong to a template. This also tells DA roughly where in terms of x, y coordinates it should look to extract a particular field.
  4. Upload a batch of documents to DA, either through the web UI, the API (which is what the DX asset uses), or by uploading the documents to cloud storage, such as AWS S3 or Google Drive.
  5. Submit the batch and specify which template you want to use. You can also let DA select the template for you. DA uses alignment features, such as lines and logos and the bounding boxes of fields that are present in the templates to find the best matching template.
  6. DA processes the documents in the background. If needed, it will choose the best matching template to use. Then it will extract the data according to the fields that were defined.
  7. (Optional) Review and correct the extracted data side by side to the actual document, through the web UI.
  8. Apply transformations to the fields, such as date formatting and capitalization.
  9. Retrieve the extracted data through the web UI or API in CSV or JSON format.

Next steps

There’s a free trial available, that you can sign up for here: https://shreddr.captricity.com/accounts/signup. Production usage requires getting a quote from a sales representative. There’s no fixed pricing, as it depends on numerous factors, such as the volume of pages that you need to process, and the SLA that you need. There’s also a Professional Services engagement that you can consider to fast-track moving a use case into production. Your sales representative can help you with this as well. More information on DA can be found on the documentation website https://bpdocs.blueprism.com/document-automation/en-us/home.htm.

Decision

Decision (not to be confused with Decision Stages) is a product that allows developers to replace complex Decision and Choice Stage logic with an ML model call. If you have complex, nested if-else branching, you can consider using Decision instead.

Figure 13.1 – Convert complex Decision and Choice Stage logic to a single Action

It’s suitable for teams that don’t have access to data scientists, and have some RPA success already, but are just getting started on their IA journey. Decision will allow them to get their feet wet with IA in some form, even if they aren’t familiar with the concepts of training a new model, deploying it, and auditing predictions.

Decision is an AutoML solution that conceals ML lifecycle complexities from users. Users simply define the potential outputs and inputs of the model through a web interface. Decision then generates random variations of the inputs and asks the user what the correct prediction should be. After a few rounds of this, an ML model is built.

Decision isn’t a standalone solution. It’s a plugin that’s installed and administered through another web-based product called Blue Prism Hub. If you’re looking to try Decision, you’ll need to have a working Hub installation as a prerequisite.

I don’t think that Decision is suitable for use cases where there are many columns of input data (> 10). The reason for this is that “training” is done one sample at a time through Hub’s web UI by someone manually looking at the inputs and telling Decision what the prediction should be. Large numbers of inputs make this process very cumbersome. I also think that Decision has to compete with a viable alternative, which is to simply create the complicated logic diagram through the Process Studio. Another point to consider is how straightforward the logic is to change. Imagine you need to add another input to the model, or add another prediction output possibility. This could lead to redoing the entire “training” process of the model, which could be extremely time consuming, since it’s done through a web UI. It might be easier to modify a Process diagram instead.

How is Decision related to IA?

Decision provides a direct way to build and deploy ML models into Production. Installing Decision requires deploying a new server, although you can cohost it on the server where Hub resides. The model runs as a web-based service on either Windows or Linux. Decision features model versioning which was an important topic featured in multiple previous chapters. There’s also a model history audit trail that keeps track of the inputs, output, confidence scores, and model version for all predictions.

The model should have tabular inputs. You can’t use it with image, audio, or prompt inputs. The output can either be numerical (regression) or categorical (classification) While the underlying model isn’t public knowledge, based on the third-party acknowledgments that reference scikit-learn and SciPy, I expect that it’s using a tree-based algorithm. Tree-based algorithms are also one of the few families of algorithms that work for both regression and classification problems, and they can be trained relatively quickly.

Using Decision

At a high level, the steps to use Decision are as follows:

  1. Create a new Decision model in Hub.
  2. Choose what type of output the model should predict. This can be either numerical or categorical (label text). If numerical, you can limit the range of numbers that are predicted. If categorical, you can define the list of labels.
  3. Define the input variables to the model by giving each an identifying name and type (numerical or categorical).
  4. Now that the inputs and output have been defined, we can add fixed rules. Fixed rules are run before the ML model is used. This allows you to have a hybrid decision-making strategy that first uses fixed, hardcoded rules, followed by an ML model.
  5. If the model output is categorical, you’ll need to manually create one data sample for each prediction label you’ve defined. Decision will loop through each of the potential prediction outputs, and you’ll be asked to create one set of inputs that leads to that prediction.
  6. After one data sample is created for each output, Decision starts to present randomized permutations of the input data, and you’re asked to select the correct prediction. This is known as the training phase of Decision. As you do more of these, the model should become more accurate. Stop when you feel that the model meets your accuracy requirements.
  7. At this point, we have a model that has been trained using multiple human-validated datapoints. Next, Decision uses the model, generates random input data, and shows you its prediction and the confidence level. You have two choices here. First, you can correct the prediction if it’s incorrect. Alternatively, you can change the input data so that it results in the prediction that Decision has been made. This is known as the calibration stage of Decision. Repeat this step as many times are you like.
  8. Save the model and make a note of the Model Id.
  9. In your BP diagram, use the Utility - Decision VBO and the Get Prediction Action, and pass in the Model Id (as well as your input data). The predicted output and confidence will be returned.

Interact

Interact is another product that’s a part of BP Hub. It’s a web-based HITL interface that enables collaboration between people (typically business users) and their BP Digital Workers. Through Interact, you can design web forms to show to business users and ask them for input. When a business user submits an Interact web form, you can do a variety of things, such as trigger a Process to start, or continue a Process that was waiting for human approval.

How is Interact related to IA?

As you’ve probably guessed, Interact is a good choice for an ML prediction review interface. It has BP Actions to push review data into an Interact form. Forms can be assigned to a specific Interact user (called a reviewer), or a group of reviewers. The reviewer can fill in the form with the correct prediction, and send it back into BP for further processing. It’s tightly integrated into the BP Enterprise product so you don’t need to use potentially brittle data-sharing interfaces such as Excel files in shared folders.

Interact will work for numerical and text-based input and output values. It won’t work if the ML inputs/output has images, as Interact can’t show images in the web UI. Interact also has its own audit trail that keeps track of who performed the review, and what their reviewed prediction was.

Using Interact

At a high level, the steps to use Interact as an interface to review ML predictions are the following:

  1. Create an Interact form. A form must be associated with a BP Process.
  2. Set the Work Queue into which reviewed prediction data should be pushed. This should be a new Queue and not the one used in the original Process. Set an SLA for the form if required. This would correspond to the HITL review SLA.
  3. Add pages to the form if you need to break your form up into multiple pages.
  4. Add the desired form fields such as text, file uploads, and tables onto the pages. All of these fields can have default values, ranges, and restrictions on what inputs can be entered.
  5. Save the form.
  6. Deploy the form by adding an Interact user role to the form. This maps Interact users to the forms that they can access and submit.
  7. Amend the Process that makes ML predictions to use the Utility – Interact API VBO. Once the Process determines that HITL review is required, the Digital Worker should retrieve the form and fill it in with the input data and prediction. Then use the Raise Submission or Raise Submission to Role Action to request someone to review the prediction. The Work Queue Item should store a reference to the _requestId Data Item returned by the Action.
  8. The Interact user (reviewer) visits the Interact web UI and corrects the submission. This creates a new Item in new the Work Queue described in step 2.
  9. Amend the Process that checks for HITL reviews. Use Utility – Interact API::Get Submission and input _requestId. Retrieve the reviewed prediction data and continue on with the Process logic.

Next steps

Interact is available to customers who have Production or Business-Critical support plans. If your company is on one of these plans, you can request for an Interact license through your sales representative. With the license ready, you can download Interact from here: https://portal.blueprism.com/product/related-products/blue-prism-interact-premise.

Installing Interact is a fairly complex process, even when we exclude the Hub parts of the equation. Further details around the infrastructure, hardware, and software requirements of Interact can be found here: https://bpdocs.blueprism.com/hub-interact/4-7/en-us/installation/install-interact.htm.

The most difficult part will be designing your forms, and the installation, both of which are covered by this offering. You can reach out to your sales representative for more information. The learning curve of the product itself isn’t so bad, but getting Interact set up from an infrastructure point of view can be a multi-month affair. More information about Interact can be found here: https://bpdocs.blueprism.com/hub-interact/4-7/en-us/home-interact.htm.

Future IA trends

I believe that three broad IA trends will play out in the coming years. The first is an accelerated integration of AI directly into the RPA product suite. Next is the widespread adoption of LLMs for generic, low-cognitive AI tasks. And the third trend is increased scrutiny of IA, for ethics and safety reasons.

Improved AI product integration

As we’ve seen in this chapter, BP (and other RPA vendors) have numerous AI-related products as part of their product suites already. But as IA starts to develop, I think that the following changes to the product offerings will occur:

  • There will be an increased push towards using the cloud-based version of products, with updated terms of service that allow vendors to capture high-level information about your process flows. Most RPA vendors will want to start training their own LLM-like models, and offer a Copilot-style of product functionality.
  • The HITL interface tools for ML review and building ML models will become more tightly integrated. In BP, this would be Interact and Decision.
  • The addition of data and ML model monitoring tools.
  • This is probably a number of years away, but expect the release of products that implement generic ML interpretability techniques such as SHAP, LIME, and ICE plots that tie into the ML audit logs automatically. These interpretability techniques would be for models that are built using the vendor’s ML product suite, such as Decision. This is because many of these techniques require very tight integration to the models themselves. Some require server log access and access to the original training data. Both this and the preceding point will be needed due to increased scrutiny on using ML in business.

Democratized ML using LLMs

Many people think that LLMs will lead to an explosion of chatbot-based IA use cases. While this may be the case, I believe that the greater value-add of LLMs is that they democratize ML, similar to how low-code development solutions have democratized parts of software development.

The advent of LLMs allows citizen developers to participate in the use of ML and develop their own IA process flows. Before LLMs, I would place ML beyond the typical citizen developer’s skill level, but that won’t be the case for very much longer. While the use cases of LLMs remain in the class of cognitive skills that can be done by junior high school students, such as data extraction and text generation, we can expect this to improve over time. As LLMs become more powerful, citizen developers will have access to increasingly powerful AI.

Despite this, there will always be reasons to custom-build ML models. When there’s potential scrutiny around your use of AI, you’d be well served to build a custom model with data and a technology stack that you’ve vetted. LLMs won’t hold up to this degree of scrutiny as the datasets needed to create them are too vast to understand.

AI ethics and safety

As mentioned in Chapter 4, governments are currently shaping their views on AI and IA, and how these relate to human workers. Universities and other policy think tanks are also throwing their opinions into the mix to help shape AI policy. For example, in November 2023, MIT published a number of government policy briefs to help shape the governance of AI. These briefs can be found here: https://computing.mit.edu/ai-policy-briefs. While these haven’t become full regulation, reading them gives us a sense of what is likely to come in the future. Here are some of the policy recommendations that are directly relevant to this book:

  • AI must be regulated, potentially by newly established federal agencies.
  • Auditing regimes of AI systems must be developed. Some high-risk use cases should require auditing before they’re put to use, while low-risk use cases could be audited retrospectively.
  • AI systems should be interpretable.
  • In terms of liability, the ultimate provider of the service should be responsible, even if the model is built on top of another foundational model.

This book was written with a broad view of future AI governance in mind. IA must happen within the bounds of the regulatory system, which is being shaped as we speak. IA team members should make a conscious effort to stay up to date with local developments in AI regulation and tune their internal practices to meet local regulatory requirements.

Summary

In this chapter, we looked at four additional BP products that can help you on your IA journey. Two of these products, Decipher IDP and Document Automation focus on processing documents using ML. Document Automation is a hosted solution that focuses on extracting data from poor-quality scanned documents with handwriting in fixed, structured forms. Decipher is a self-hosted solution that builds ML models specific to your document types. It can classify documents and extract data from structured, semi-structured, and unstructured documents.

Decision is an AutoML product that hides the complexities of ML away from the users. It can be used to replace complex Decision and Choice Stage logic with a single ML call. Users can quickly create models through Decision’s web interface on Hub. First, specify what the predicted output should be. Then define the inputs. Decision then presents the user with different combinations of inputs, and asks them to provide the correct prediction. A model will be built after repeating this a few times. Decision has built-in IA best practices, such as ML versioning and audit trails.

Interact is BP’s solution to human-robot collaboration. Through Interact, you can define web forms that are presented to users (reviewers) through the Hub platform. Users can interact with BP Digital Workers in numerous ways through these forms. For example, they can trigger a Process to start, or act as a reviewer for ML predictions. SLAs for the reviews can also be defined.

Finally, we discussed three future IA trends that I believe will happen in the next few years. First, there will be many improvements to vendor product offerings to address IA-specific concerns. Next, LLMs will greatly expand the number of IA use cases, as citizen developers are enabled to use ML in their business processes. And third, regulations around the use of AI will start to change our IA practices as they become better defined.

Appendix

IA Risk Management

This appendix summarizes a portion of my IA thesis research. A primary focus of my research was to elicit the risks that IA entails to firms. The motivation behind this is to refine the industry’s understanding of this under-studied and under-appreciated topic, which I believe is critically important to the global adoption and success of IA. I also hope that understanding the risks of IA will force us to consider how to address those risks and reduce the negative effects that automation can have on society.

My study revealed 36 IA risks. These risks were categorized into two main groups: socio-organizational and operational. Socio-organizational risks stem from the relationships between different social constructs. This could, for example, be between firms and the legal system, or management and employees. Operational risks are those that appear during ongoing IA operations. They’re narrower in scope compared to socio-organizational risks, and are project-based or technical in nature.

I also looked at what mitigation measures exist to combat the IA risks. These risk mitigation techniques have guided the development of this book and have been implemented into the reusable IA templates, where possible. You can take advantage of numerous risk-reducing measures simply by using the templates as a starting point for your IA solution designs.

Socio-organizational IA risks

Four organizational constructs were identified. They are the environment, the enterprise, third parties, and employees. Conflicts between these groups led to 22 IA risks.

Figure Appendix.1 – The four organizational groups and how risks emerge between them

A summary of the 22 risks can be found in the following table:

Socio-Organizational IA Risks (22)

Environmental Risks (3)

Description

Compliance

New regulations around the use of AI in business are being written as we speak. Not only must we comply, but third-party ML vendors must also be in compliance. Current regulations are for the most part quite recent, with the laws themselves sometimes being vague.

Ethics

Both algorithms and the data used to train them can be biased. We should avoid using protected data, such as ethnic origin, age, or gender, unless it’s explicitly allowed. Features can also be correlated with sensitive data; for instance, a geographic region can be associated with certain ethnic groups. Another aspect of ethics is the firm’s treatment of employees who are affected by IA.

Regulatory

Future laws might require businesses to completely disable algorithmic decision-making and to offer human-only intervention in cases. A prudent IA team will be risk-averse and implement ways to disable ML in IA, even if laws that require them don’t exist yet. Explainable AI should also be explored.

Enterprise Risks (5)

Description

Departmental Resistance

Managers fear that IA will narrow their sphere of influence due to budget and headcount reductions. This can lead to department-wide non-cooperation of IA initiatives.

Employee Turnover

IA adoption may lead to high employee turnover. Reasons can include a reduction in working hours, increased stress, and ideological opposition.

Financial Loss

Work that’s completed based on an incorrect ML prediction and missed ML SLAs can lead to financial loss. This could be from legal damages or the incorrect work that is performed following an incorrect prediction.

Information Asymmetry

IA can create power imbalances within a firm. Business units that use IA will presumably have more access to data, and be able to complete work faster and more accurately. This can exacerbate issues in firms that have a competitive culture.

Loss of Control

The use of third-party vendors for ML services can lead to a loss of control. Models can be updated without prior notice, and services can have unplanned outages, preventing you from meeting SLAs.

Employee Risks (8)

Description

Cognitive Work Overload

Simple cognitive tasks are more frequently automated than complicated ones. Over time, many simple cognitive worker tasks might be automated away, leaving only the difficult ones for people to work on. This can lead to increased job stress and cognitive overload.

Loss of Job Meaning

Work that’s meaningful to employees might be the target of automation. If automated away, people’s job satisfaction can decrease.

Loss of Job Security

IA can automate knowledge-based tasks. This will lead to more people being worried about their jobs.

Mistrust in Management

IA can foster mistrust between management and those who are affected by IA.

Mistrust in Model Predictions

People may believe that an ML model can’t replicate their knowledge. They might not trust the results of the model. This can lead to the active resistance or sabotage of IA endeavors. Staff might also decide to complete cases manually and not use the IA process at all.

Prediction Accountability

The company should explicitly define who is responsible for the outcomes of incorrect ML predictions.

Reduced Work Preparedness

Staff spend less time overall looking at the details of work cases that are heavily automated. More effort needs to be spent getting up to speed if the case needs to be manually handled. This could be important for time sensitive work cases, such as customer service calls.

Worker Deskilling

Decision-making is a skill that requires use to maintain. If human decision-making is automated, workers’ skill levels may decline due to being used less often. In the extreme case, this could happen across multiple workers, leading to the permanent loss of decision-making skill for a particular function.

Third-Party Risks (6)

Description

Assignment of Liability

When multiple parties are involved in IA, such as a services firm and a cloud vendor, it can become unclear who is liable for damages if something goes wrong in an IA process. It can also be impossible to prove that one of the third parties is at fault.

Attract Competitive Response

Promoting the use of IA publicly can lead to actions from competing companies. For example, competition could start to invest in their own IA capabilities, or use this information to portray your firm in a negative light.

Conflicts of Interest

ML vendors that have developed an algorithm for you could resell your algorithm to others, or raise the price significantly after it has become critical to your operations.

Missed Servicing Opportunities

When we use AI to interface with customers (chatbots), it can only form responses based on what it has previously seen. Edge cases and special requests that a customer has can’t be serviced, where they could be if a human was part of the interaction.

Performance Agreement Breaches

IA might require existing contractual performance agreements to be renegotiated. It introduces additional points of failure to the automated process, such as server infrastructure, which can cause service downtime, and result in missed SLAs.

Reputation Loss

Work that’s performed incorrectly due to a wrong ML prediction can cause reputational loss to the company.

Description

Control Flow Drifts

Changes to the business process or logic paths could cause us to rebuild the ML model. Imagine if we want to handle a new category of cases and the type of case is determined by a classification model. This means that the model needs to predict a new label, necessitating redevelopment.

Difficult to Carry out Error Detection

Automating based on incorrect ML predictions can lead to errors in downstream processing. Finding these errors can be difficult.

Reduced Understanding of Business Logic

Institutional knowledge of a business process can be lost especially if higher-order human decision-making is replaced by ML models.

Time Lag Effects

Downstream work that follows an incorrect ML prediction can lead to lots of incorrect processing. The gap between when the incorrect prediction is made, when it’s discovered, and all of the incorrect work that’s done in between is known as “time lag effects.”

ML Model Risks (3)

Description

Adversarial Attacks

ML algorithms can be broadly subjected to adversarial attacks. This is a new attack vector that’s present when ML is used. This type of attack on the IA solution’s ML model could lead to undesired work being performed in the automation.

Performance Degradation

The predictive accuracy of models is broadly known to reduce as time passes. Models need to be actively managed and refreshed to counteract this. Models and technologies can be made obsolete, or even be discontinued due to newer techniques and libraries.

Transfer Learning Bias

Transfer learning is when an existing ML model is used as a base, with your customized model built on top of it. This is very popular, especially with LLMs. However, any latent issues with the base model will also be present in the customized model. You might be unable to address the issues unless you stop using the base model completely.

Data Risks (4)

Description

Data Bias

Models may perform poorly on real-life data due to biases in the data. Types of data bias include historical bias, which reinforces stereotypes of particular groups; representation bias, where certain populations are underrepresented in collected data; measurement bias, where proxy data is collected (for example, using the number of police arrests as a proxy for crime rate); and aggregation bias, where a catch-all ML model is developed when multiple models should have been used instead.

Data Drift

The input data that was used to train a model might be outdated and no longer representative of input data today. This would make the model unfit for use against current data.

Data Privacy & Security

Building some ML models is out of reach for businesses, due to a lack of expertise, GPU and CPU compute resources, or budget. Sending data to third-party models leads to data privacy concerns, and increases the chances of data leaks.

Data Quality

Low data quality leads to models with poor performance. “Quality” is an ill-defined concept, but typically includes the number of data samples, having the proper data structure, a lack of noise or errors, having “complete” data, and having highly relevant features.
